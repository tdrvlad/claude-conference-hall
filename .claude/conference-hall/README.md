# Conference Hall — the service

This directory is a small, self-contained **control plane** for a
manager/subagent Claude Code setup. It is two programs plus a web page:

| File | What it is |
|------|------------|
| `server.py`         | The **feed service** — an HTTP + SQLite store of feed items. The single source of truth. |
| `mcp_server.py`     | An **MCP server** that wraps the feed as 7 first-class tools Claude can call. A thin proxy over `server.py`. |
| `static/index.html` | The **Conference Hall UI** — a self-contained dark dashboard the human watches in a browser. |
| `mcp_launch.sh`     | Portable launcher Claude Code runs to start `mcp_server.py` (see *Portability* below). |
| `run.sh`            | Starts the feed service (detached, idempotent). |
| `requirements.txt`  | Python deps: fastapi, uvicorn, httpx, mcp[cli]. |
| `.venv/`            | Created by `../../setup.sh`. *(gitignored)* |
| `conference_hall.db`| SQLite db, created on first run. *(gitignored)* |

---

## The idea in one paragraph

One Claude session is a **manager** that talks to you and delegates work to
**background subagents**. Every actor — the manager and each subagent — writes
to its **own channel** in this shared feed. The web UI shows one tab per
channel plus an "All" tab, with decisions the human must make pinned to the
top. Because each actor only ever writes its own channel and the server
serializes every write, there is no file-clobbering — this replaces the old
pattern of agents fighting over a shared scratch file. It is a **current-state
dashboard, not an append-forever log**: resolved items get *archived* (hidden
from the live view, kept on disk), which is what keeps it bounded.

```
   manager (Claude)  ─┐
   subagent A        ─┤  MCP tools        HTTP            ┌─ writes ─┐
   subagent B        ─┼─► mcp_server.py ──────► server.py ─► SQLite  │
   (each via curl)   ─┘   (proxy)              (feed API)  └──────────┘
                                                   ▲
                                  browser polls ───┘  (static/index.html)
                                  every 2.5s          the human watches
```

## The data model

Every item is one row:

```
{
  id:       "it_<hex>",     # generated by the server
  channel:  "manager",      # the actor that wrote it (= its name)
  type:     "decision",     # see types below
  html:     "<b>…</b>",     # arbitrary, trusted, rich HTML — rendered raw
  pinned:   true,           # pinned items sort above the rest
  archived: false,          # archived items drop out of the live view
  created:  1782317816.38   # epoch seconds
}
```

**Channels** partition writers. `"manager"` is the main session; each subagent
uses its own `name` as its channel. One writer per channel.

**Types** (the UI colour-codes the badge):

| type | meaning | colour |
|------|---------|--------|
| `decision` | the human must choose — **auto-pinned**; archive when resolved | red |
| `todo`     | an overarching task list — **auto-pinned**; updated in place | amber |
| `result`   | a finished deliverable / outcome | green |
| `spawn`    | the manager dispatched a subagent | cyan |
| `handoff`  | the manager routed one agent's need to another | purple |
| `status`   | started / progress / done | blue |
| `note`     | anything else (default) | grey |

**Pinning** — `decision` and `todo` auto-pin (unless you pass `pinned`
explicitly). Pinned items sort first; within pinned/unpinned, newest first.

**Archiving = the bounding mechanism.** Nothing rotates or expires. When a
decision is answered or a task is done, you `archive` it: it leaves the live
feed but stays on disk (`include_archived=true` still returns it). The panel
therefore always reflects *current* state and never grows without bound.

## HTTP API (`server.py`)

Base URL `http://127.0.0.1:8787` (localhost only, no auth — personal tool).

| Method & path | Body / query | Does |
|---------------|--------------|------|
| `POST /items` | `{channel, html, type="note", pinned?}` | Create an item. Auto-pins `decision`/`todo` unless `pinned` given. Returns `{id, channel, type, pinned}`. |
| `GET /items`  | `?channel=&include_archived=false` | List items, ordered `pinned DESC, created DESC`. Omit `channel` for all channels. |
| `GET /channels` | — | `[{channel, n, decisions}]` — live item count + open-decision count per channel. Drives the tabs/badges. |
| `PATCH /items/{id}` | `{html?, type?, pinned?}` | Partial update in place. `400` if the body is empty, `404` if no such id. |
| `DELETE /items/{id}` | — | Hard delete. `404` if missing. |
| `POST /items/{id}/archive` | — | Soft-remove (`archived=1`). `404` if missing. |
| `POST /channels/{channel}/clear` | — | Archive every live item in a channel (soft reset of a tab). |
| `GET /` | — | Serves the UI (`static/index.html`). |

Config via env: `FEED_PORT` (default `8787`), `FEED_DB` (default
`conference_hall.db` next to `server.py`).

### Talk to it with curl (the fallback path)

Agents have Bash, so curl always works even without the MCP tools:

```bash
# post a decision (auto-pins)
curl -s -X POST http://127.0.0.1:8787/items \
  -H 'content-type: application/json' \
  -d '{"channel":"manager","html":"<b>Postgres or SQLite?</b>","type":"decision"}'

# read a channel
curl -s 'http://127.0.0.1:8787/items?channel=manager'
```

## MCP tools (`mcp_server.py`) — the preferred path

`mcp_server.py` exposes the feed as the MCP server **`conference-hall`** with
7 tools. It is a *proxy*: it forwards to the running `server.py`, so **the feed
service must already be up** or every tool errors with a connection refusal.

| tool | wraps | use it to |
|------|-------|-----------|
| `post_item(channel, html, type="note", pinned=None)` | `POST /items` | log to *your own* channel |
| `list_items(channel=None, include_archived=False)`   | `GET /items` | read current state before deciding |
| `list_channels()`                                     | `GET /channels` | see who has open decisions |
| `update_item(id, html=None, type=None, pinned=None)`  | `PATCH /items/{id}` | refresh a TODO / unpin a resolved decision |
| `archive_item(id)`                                    | `POST …/archive` | resolve an item (preferred over delete) |
| `delete_item(id)`                                     | `DELETE /items/{id}` | remove a mistake |
| `clear_channel(channel)`                              | `POST …/clear` | soft-reset a tab |

Each tool has a detailed docstring — Claude reads those to know when/how to
call it. The MCP server speaks **stdio** and is launched by Claude Code, not by
you (see below).

## How the wiring fits together

1. **`server.py` must be running.** The root `CLAUDE.md` tells the manager to
   start it on its first turn (`bash .claude/conference-hall/run.sh`); you can
   also start it yourself. It binds `127.0.0.1:8787`.
2. **Claude Code launches the MCP server** from the project-root `.mcp.json`.
   That entry runs `bash ./.claude/conference-hall/mcp_launch.sh`, which
   self-locates and execs `.venv/bin/python3 mcp_server.py`.
3. **`mcp_server.py` proxies** every tool call to `server.py` over HTTP.
4. **The browser** (`static/index.html`) polls `GET /channels` and `GET /items`
   every ~2.5s and renders each item's `html` raw, so tables / `<pre>` / `<svg>`
   display. It persists the active tab in `localStorage` and shows a clear
   "service unreachable" state if the feed is down.

> **Why the launcher?** Claude Code's `.mcp.json` needs a command it can run
> from the project root. Hard-coding an absolute venv path makes the repo
> non-portable (every clone is at a different path). `mcp_launch.sh` resolves
> its *own* location via `BASH_SOURCE`, so `.mcp.json` only needs the
> project-root-relative path to the script — the same committed file works in
> any clone. If relative resolution ever fails on your machine, run
> `./setup.sh --absolute-mcp` to regenerate a per-machine absolute `.mcp.json`
> from `.claude/.mcp.json.example`.

> **Why `.mcp.json` is at the repo root (not in `.claude/`).** Claude Code only
> auto-discovers project MCP servers from a `.mcp.json` at the project root —
> a `.claude/.mcp.json`, or an `mcpServers` block inside `.claude/settings.json`,
> is **not** loaded (verified). So the one functional file stays at root; only
> the optional `.mcp.json.example` template lives under `.claude/`.

## The SubagentStop hook

`../hooks/subagent_stop.py` (registered in `../settings.json`) posts a
`status` "finished" item to a finishing subagent's channel, so its tab flips to
done without anyone polling. It figures out the channel from the hook payload's
`subagent_name` if present, otherwise by scanning the subagent's transcript for
the last channel it posted to. It is **fail-silent** — if the service is down
or the channel can't be determined, it does nothing and exits 0. The
authoritative "done" signal is still the agent's own `result` post.

## Running standalone

```bash
../../setup.sh                       # once: builds .venv, installs deps
bash run.sh                          # start detached on :8787
bash run.sh --foreground             # or run in this terminal
FEED_PORT=9000 bash run.sh           # custom port
fuser -k 8787/tcp                    # stop it
```

Then open <http://127.0.0.1:8787/>.

## Why it's safe to render HTML raw

The feed is **localhost-only, single-user, and all content originates from your
own Claude session.** There is no untrusted input and no network exposure, so
the UI renders item `html` verbatim — which is the whole point: rich tables,
diagrams, and code render correctly. Do **not** expose this service to a
network or multiple users without adding sanitization and auth.
